# 存储架构

> 状态：V1 现状 + IDB 迁移进行中。仓库当前分两层——**chrome.storage**（默认 / 兜底）和 **IndexedDB**（迁移后启用）。任何修改此层的人请读完本文。

## 1. 分层概览

```
┌─────────────────────────────────────────────────────────────────────┐
│ 调用方（~98 处）                                                      │
│   settings-controller / operations/* / sync-service / domains / ...    │
└─────────────────────────────────────────────────────────────────────┘
              │  全部经 storage API（saveWallet / getAccounts / getNetworks / ...）
              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ storage API 层（js/storage/wallet-storage.js 等）                      │
│   按 getStorageBackend() 分发：'chrome' → chrome.storage  / 'idb' → IDB │
└─────────────────────────────────────────────────────────────────────┘
        │                                    │
        ▼                                    ▼
┌─────────────────────┐            ┌─────────────────────────────────┐
│ chrome.storage.local │            │ IndexedDB（yeying_wallet_db）     │
│                     │            │   stores:                          │
│ 保留：               │            │     wallets    (keyPath id)        │
│  - user_settings     │            │     accounts   (keyPath id, idx  │
│  - selectedAccount   │            │                   by_wallet)     │
│  - selectedNetwork   │            │     networks   (keyPath chainId)  │
│  - permissions       │            │     transactions (keyPath hash)   │
│  - contacts          │            │                                   │
│  - settings          │            │ 保留源数据：迁移成功后              │
│  - ucan              │            │ chrome.storage 三个 key           │
│  - mpc               │            │ (wallets/accounts/networks)        │
│                     │            │ 仍保留（双版本回退期）              │
│ 迁移集：             │            │                                   │
│  - wallets (Map)     │            │                                   │
│  - accounts (Map)    │            │                                   │
│  - networks (Array)  │            │                                   │
└─────────────────────┘            └─────────────────────────────────┘
```

**当前后端**：`storageBackend` 内存变量，default `'chrome'`。background 启动时调 `initStorageBackend()` 跑一次性迁移，**仅迁移成功**才切 `'idb'`。

## 2. 各集合存储位置（事实表）

| 集合 | chrome.storage | IndexedDB | 备注 |
| --- | --- | --- | --- |
| `wallets` | ✅ 主源（保留） | ✅ 迁移后写 | 含加密助记词；高风险 |
| `accounts` | ✅ 主源（保留） | ✅ 迁移后写 | 含加密私钥；高风险 |
| `networks` | ✅ 主源（保留） | ✅ 迁移后写 | 元数据；可重建 |
| `user_settings` | ✅ 唯一 | ❌ | 单 key；含 backupSync*、password-cache |
| `current_account_id` | ✅ 唯一 | ❌ | 单标量；popup-controller 监听 |
| `selectedNetwork` | ✅ 唯一 | ❌ | 单标量；保留 chrome.storage.onChanged 监听 |
| `connected_sites` | ✅ 唯一 | ❌ | |
| `contacts` | ✅ 唯一 | ❌ | sync-service 通过 `onMutation` 监听 |
| `permissions` / `settings` / `ucan` / `mpc` | ✅ 唯一 | ❌ | |

`transactions` **不在 chrome.storage**——独立 IndexedDB store（v1 已有），是唯一迁移前就 IDB 的集合。

## 3. 后端开关与迁移状态机

**位置**：`js/storage/backend.js`

```
                    background 启动
                         │
                         ▼
              initStorageBackend(deps)
                         │
                         ▼
              getFlag() ─── 已设？── 是 ──→ return 'idb'
                 │  否
                 ▼
              readChrome() → writeIdb() → verify() → setFlag(true) → return 'idb'
                 │ 任一异常 / 校验不符
                 ▼
              onError() 记诊断 → 保持 'chrome'，下次重试
```

- 迁移 IO 依赖注入（`readChrome` / `writeIdb` / `verify`），便于 `tests/storage-backend.test.mjs` 在无真实 IDB 的 Node 环境验证状态机（已实现）。
- 失败**绝不抛错**（外层 `try/catch`），background 启动不因 IDB 异常中断。
- `storageBackend` 存于 `backend.js` 模块作用域，进程内一致。

## 4. 双版本共存的"灾难回退"保证

**迁移成功后，chrome.storage 三个 key（`wallets` / `accounts` / `networks`）**仍保留**，不删除。理由：

- IDB 在我（自动化）端无法实测；一旦 IDB 出问题（用户环境异常、版本升级、Store 损坏），把 `storageBackend` 改回默认 `'chrome'` 即可从 `chrome.storage` 读到全部数据。
- 删除窗口设想的"清理"**必须**等下一个大版本有充分用户数据（确认 IDB 在生产稳定）才能做。

**实际工程含义**：
- ❌ 不要在 `clearAllData()` 之外手动 `chrome.storage.remove('wallets' / 'accounts' / 'networks')`
- ❌ 不要把 `migrationCollectionsToIdb` 的 `setFlag(true)` 视为可重复运行
- ✅ `clearAllData()`（重置钱包）才同时清 chrome.storage + IDB（`js/storage/index.js:251`）

## 5. 写函数的通知机制

迁移后 IDB 写不再触发 `chrome.storage.onChanged`，**依赖**这套监听的代码会静默失效。

为解决，`js/storage/mutation-events.js` 提供内存 pub/sub：

```js
// 写函数后（两套后端都调）
notifyMutation('wallets' | 'accounts' | 'networks');

// sync-service 等订阅
onMutation((collection) => {
  if (collection === 'wallets' || ...) this.markDirty('storage-change');
});
```

- 当前订阅者：`sync-service`（用于 WebDAV 备份脏检测）。见 `sync-service.js:_attachStorageListener`。
- `onStorageChanged` **仍保留**给 `contacts` 等未迁移集合的脏检测。
- ⚠️ 新增对迁移集合的 chrome.storage.onChanged 监听**无效**——必须订阅 `onMutation`。

## 6. `clearAllData` 行为（reset wallet 路径）

`js/storage/index.js:245`：清 chrome.storage 全部 + 三个 IDB store。`js/background/operations/wallet.js:745`（`handleResetWallet`）调用之。

任何"清 IDB 但保留 chrome"的中间态会留下不可见数据——避免。

## 7. `changePassword` 的原子写

`js/background/operations/wallet.js`：
- chrome 后端：单次 `setStorage({ wallets, accounts })` 原子提交。
- idb 后端：`runMultiStoreTransaction(['wallets', 'accounts'], 'readwrite', ...)` 跨 store 单事务。

任一后端失败 → 整个事务回滚，不留新旧密码混杂状态。`tests/storage-idb.test.mjs` 验证了 IDB 端回滚路径。

## 9. 待办（不在 V1 修复窗口内）

- **chrome.storage 源数据清理窗口**：待 IDB 在生产稳定（一个完整版本周期后）才能清 `wallets` / `accounts` / `networks` 三个 key。届时设 `storageBackend: 'chrome'` 也不会丢数据。
- **`.idb` 后端下 `getMapItem` 等 chrome 专属语义**：当前每个 `isIdb()` 分发都重写 IDB 版。考虑统一封装 `StorageBackend` 接口减少重复。
- **`contacts` / `settings` / `ucan` / `mpc` 是否也迁 IDB**：待评估数据规模（当前 < 100KB/账户，影响小）。
- **`onStorageChanged` 在 chrome 后端下的兼容性**：IDB 路径下 chrome.storage.onChanged 永不触发已迁集合的变更；若未来切回 chrome 后端（灾难回退）需注意脏检测仍可用。
